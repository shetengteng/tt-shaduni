# Docs 组件示例展示方案对比

> 核心问题：UniApp 组件使用 `<view>`、`<text>` 等平台标签，**必须经过 UniApp 编译才能运行**，无法直接在标准浏览器中使用。

## 方案一览

| # | 方案 | 真实组件 | 部署难度 | 构建产物大小 | 交互性 | 维护成本 |
|---|------|---------|---------|------------|--------|---------|
| A | iframe 嵌入预构建 H5 | ✅ 完全真实 | 低（纯静态） | ~500KB | 完全交互 | 中 |
| B | 外部部署 + 链接/iframe | ✅ 完全真实 | 中（需额外部署） | 0（docs 内） | 完全交互 | 低 |
| C | HTML+CSS 视觉模拟 | ❌ 模拟 | 最低 | ~20KB | 有限 JS 模拟 | 高 |
| D | 截图/GIF/视频 | ❌ 静态 | 低 | 图片大小 | 无 | 高（需手动更新） |
| E | Vitepress + 组件编译 | ⚠️ 需适配 | 高 | 取决于 | 完全交互 | 高 |

---

## 方案 A：iframe 嵌入预构建 H5（当前实现）

**原理**：`npm run docs:playground` 将 playground 编译为纯静态文件到 `docs/playground/`，docs 的 phone 框通过 `<iframe>` 加载。

**GitHub Pages 工作流程**：
1. 本地运行 `npm run docs:playground`
2. 生成的 `docs/playground/` 目录包含 `index.html` + `assets/`，纯静态
3. 提交到 git → GitHub Pages 直接托管，**不需要启动任何服务器**
4. iframe 加载的是同域静态文件，和加载图片一样

**优点**：
- 展示完全真实的组件（包括动画、交互、状态）
- 纯静态部署，GitHub Pages 直接支持
- 组件更新后重新 `build:h5` 即可同步

**缺点**：
- 构建产物 ~500KB 需提交到 git
- 修改组件后需重新构建才能更新 docs
- iframe 与 docs 主页面的主题/语言同步需要 postMessage

---

## 方案 B：外部部署 playground

**原理**：playground 单独部署到 Vercel/Netlify/Cloudflare Pages，docs 通过链接或 iframe 引用。

**工作流程**：
1. playground 仓库配置 CI/CD 自动部署
2. docs iframe 指向 `https://tt-shaduni-playground.vercel.app`
3. 或提供 "Open in Playground" 按钮跳转

**优点**：
- docs 仓库不包含 playground 构建产物
- playground 更新自动部署
- 可独立访问完整 playground

**缺点**：
- 跨域 iframe 可能有限制
- 需要额外的部署配置
- 依赖外部服务可用性

---

## 方案 C：HTML+CSS 视觉模拟（之前的实现）

**原理**：用原生 HTML + CSS 手写组件外观，配合少量 JS 模拟交互。

**优点**：
- 零构建依赖，最简单
- 文件小，加载快
- 不依赖 UniApp 运行时

**缺点**：
- **不是真实组件**，外观可能与实际不一致
- 每个组件需要手写 HTML/CSS/JS 模拟
- 组件更新后需要手动同步模拟代码
- 复杂交互（动画、手势、弹窗）难以模拟

---

## 方案 D：静态截图/GIF

**原理**：在 playground 中截图或录制 GIF，嵌入 docs。

**优点**：
- 最直观
- 无构建依赖

**缺点**：
- 完全无交互
- 组件更新需重新截图
- 图片文件增加仓库体积
- 无法展示暗色模式等切换效果

---

## 方案 E：Vitepress + 组件编译适配

**原理**：将 UniApp 组件适配为纯 Web 版本，在 Vitepress 中直接使用。

**优点**：
- 文档系统强大（Markdown + Vue 组件）
- 组件代码和文档紧密关联

**缺点**：
- UniApp 组件使用平台标签，**需要编写 Web 适配层**
- 本质上是维护两套组件（UniApp + Web）
- 工作量大，不推荐

---

## 建议

**对于 UniApp 组件库，方案 A（iframe 嵌入预构建 H5）是最务实的选择**：

- 它是唯一能以零额外代码展示真实组件的方案
- 部署是纯静态的，GitHub Pages 完全支持
- 构建步骤简单：`npm run docs:playground` 一条命令

**如果不想在 git 中提交 playground 构建产物**，可以：
1. 在 `.gitignore` 中忽略 `docs/playground/`
2. 在 GitHub Actions CI 中构建（推荐）：
   ```yaml
   - run: cd playground && npm install && npm run build:h5
   - run: cp -r playground/dist/build/h5 docs/playground
   ```

**如果想完全避免 iframe**，则只能选择方案 C（视觉模拟）或方案 D（截图），但两者都不是真实组件。
