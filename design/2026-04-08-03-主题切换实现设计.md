# TT-UI 主题切换实现设计

> 分析 UniApp 多端环境下主题 / 暗黑模式切换的技术方案与实现路径

---

## 1. 问题定义

在 Web 端，主题切换只需在 `:root` 或 `<html>` 上切换 class 即可。但在 UniApp 多端环境下面临以下挑战：

| 挑战 | 说明 |
|------|------|
| **无全局根 DOM** | UniApp 的 App.vue 不渲染视图，每个 page 是独立根节点 |
| **小程序无 :root** | 小程序使用 `page` 选择器代替 `:root`，且不支持 `<html>` 属性 |
| **组件样式隔离** | 小程序自定义组件默认样式隔离，外部 CSS Variables 无法穿透 |
| **页面切换断开** | 每个页面是独立实例，A 页面的主题状态不会自动同步到 B 页面 |
| **provide/inject 跨页** | UniApp 的 provide/inject 不跨页面，App.vue 的 provide 无法在 page 中 inject |

---

## 2. 业界方案对比

### 2.1 方案 A：每页根节点绑定 style（手动方案）

```vue
<!-- 每个页面 -->
<template>
  <view :style="themeVars" class="page">
    <!-- 页面内容 -->
  </view>
</template>

<script setup>
import { useThemeStore } from '@/stores/theme'
const themeStore = useThemeStore()
const themeVars = computed(() => themeStore.cssVarsString)
</script>
```

| 优点 | 缺点 |
|------|------|
| 简单直接 | 每个页面都要手动绑定 |
| 完全控制 | 忘记绑定则该页面无主题 |
| 兼容所有平台 | 代码冗余度高 |

### 2.2 方案 B：ConfigProvider 组件（wot-design-uni 方案）

wot-design-uni 的实现方式：

```vue
<wd-config-provider :theme="theme" :theme-vars="themeVars">
  <page-content />
</wd-config-provider>
```

**原理**：ConfigProvider 内部通过 `:style` 在自身节点上设置 CSS Variables，子组件通过 `var()` 引用。

```typescript
// ConfigProvider 核心逻辑（简化）
const rootStyle = computed(() => {
  const vars: Record<string, string> = {}

  // 主题预设变量
  if (props.theme === 'dark') {
    Object.assign(vars, darkThemeVars)
  }

  // 用户自定义变量
  if (props.themeVars) {
    Object.assign(vars, props.themeVars)
  }

  return Object.entries(vars)
    .map(([key, val]) => `--${kebabCase(key)}: ${val}`)
    .join(';')
})
```

| 优点 | 缺点 |
|------|------|
| 声明式 API，直觉清晰 | 仍需每个页面包裹 |
| 支持局部主题覆盖 | provide/inject 不跨页面 |
| TypeScript 类型提示 | 组件样式隔离问题 |

### 2.3 方案 C：@uni-ku/root 虚拟根组件

通过 Vite 插件在编译时为每个页面注入一个虚拟根组件：

```vue
<!-- App.ku.vue — 全局根组件 -->
<template>
  <tt-config-provider :theme="currentTheme">
    <KuRootView />
  </tt-config-provider>
</template>

<script setup>
import { useTheme } from './composables/use-theme'
const { currentTheme } = useTheme()
</script>
```

| 优点 | 缺点 |
|------|------|
| 真正的全局根组件 | 引入额外依赖 |
| 一处配置，全页生效 | 编译时注入，调试可能不直观 |
| 配合 ConfigProvider 完美 | 对 UniApp 版本有兼容要求 |

### 2.4 方案 D：page 选择器 + Pinia 状态同步

利用小程序 `page` 选择器定义变量，通过 Pinia 全局状态在 `onShow` 时同步：

```css
/* light 主题 */
page {
  --tt-background: #ffffff;
  --tt-foreground: #0a0a0a;
}

/* dark 主题 */
page.dark {
  --tt-background: #0a0a0a;
  --tt-foreground: #fafafa;
}
```

```typescript
// 每个页面的 onShow 中同步主题
onShow(() => {
  const theme = useThemeStore().theme
  // 通过 uni API 设置页面根节点 class
  // ⚠️ 小程序 page 元素无法直接操作 class
})
```

| 优点 | 缺点 |
|------|------|
| 利用原生 page 元素 | 小程序无法动态修改 page 的 class |
| CSS 层面实现 | 需要 hack 方式注入 |
| 性能最优 | H5 / App / 小程序行为不一致 |

### 2.5 方案 E：微信官方 DarkMode 配置

微信小程序原生暗黑模式支持：

```json
// app.json
{
  "darkmode": true,
  "themeLocation": "theme.json"
}
```

```json
// theme.json
{
  "light": {
    "navigationBarColor": "#000000",
    "backgroundColor": "#ffffff"
  },
  "dark": {
    "navigationBarColor": "#ffffff",
    "backgroundColor": "#000000"
  }
}
```

| 优点 | 缺点 |
|------|------|
| 官方方案，最佳兼容 | 仅支持跟随系统，不支持手动切换 |
| 自动处理导航栏等系统 UI | 只能配置有限的系统级变量 |
| 零 JS 运行时开销 | 无法自定义业务组件的主题变量 |

## 3. 推荐方案：分层主题架构

综合以上方案，TT-UI 采用 **三层架构**：

```
┌─────────────────────────────────────────┐
│  Layer 3: 微信 DarkMode 配置 (系统 UI)     │
│  处理：导航栏、TabBar、系统弹窗颜色         │
├─────────────────────────────────────────┤
│  Layer 2: ConfigProvider + CSS Variables  │
│  处理：组件库内部所有组件样式               │
├─────────────────────────────────────────┤
│  Layer 1: Pinia 全局状态 + 持久化          │
│  处理：主题状态管理、跨页面同步、存储        │
└─────────────────────────────────────────┘
```

### 3.1 Layer 1 — 主题状态管理

```typescript
// composables/use-theme.ts
import { ref, computed, watch } from 'vue'

type ThemeMode = 'light' | 'dark' | 'system'

const STORAGE_KEY = 'tt-ui-theme'

const themeMode = ref<ThemeMode>('system')
const systemTheme = ref<'light' | 'dark'>('light')

// 初始化：从本地存储恢复 + 监听系统主题
function initTheme() {
  const stored = uni.getStorageSync(STORAGE_KEY) as ThemeMode
  if (stored) {
    themeMode.value = stored
  }

  // 获取系统当前主题
  const sysInfo = uni.getSystemInfoSync()
  systemTheme.value = sysInfo.theme || 'light'

  // 监听系统主题变化
  uni.onThemeChange?.((res: { theme: string }) => {
    systemTheme.value = res.theme as 'light' | 'dark'
  })
}

// 实际生效的主题
const resolvedTheme = computed<'light' | 'dark'>(() => {
  if (themeMode.value === 'system') {
    return systemTheme.value
  }
  return themeMode.value
})

// 切换主题
function setTheme(mode: ThemeMode) {
  themeMode.value = mode
  uni.setStorageSync(STORAGE_KEY, mode)
}

// 切换 light ↔ dark
function toggleTheme() {
  setTheme(resolvedTheme.value === 'light' ? 'dark' : 'light')
}

export function useTheme() {
  return {
    themeMode,
    resolvedTheme,
    isDark: computed(() => resolvedTheme.value === 'dark'),
    setTheme,
    toggleTheme,
    initTheme
  }
}
```

### 3.2 Layer 2 — ConfigProvider 组件

```vue
<!-- components/tt-config-provider/tt-config-provider.vue -->
<template>
  <view :class="rootClass" :style="rootStyle">
    <slot />
  </view>
</template>

<script setup lang="ts">
import { computed, provide, toRefs } from 'vue'
import { useTheme } from '../../composables/use-theme'
import { lightTokens, darkTokens } from '../../styles/tokens'
import type { ConfigProviderProps } from './props'

const props = withDefaults(defineProps<ConfigProviderProps>(), {
  theme: undefined,
  themeVars: () => ({})
})

const { resolvedTheme } = useTheme()

// 优先使用 props.theme，否则用全局 resolvedTheme
const activeTheme = computed(() => props.theme ?? resolvedTheme.value)

const rootClass = computed(() => [
  'tt-config-provider',
  `tt-config-provider--${activeTheme.value}`
])

const rootStyle = computed(() => {
  const baseTokens = activeTheme.value === 'dark' ? darkTokens : lightTokens

  // 合并用户自定义变量
  const merged = { ...baseTokens, ...props.themeVars }

  return Object.entries(merged)
    .map(([key, val]) => `--tt-${camelToKebab(key)}: ${val}`)
    .join(';')
})

// 提供给子组件（同页面内有效）
provide('tt-theme', activeTheme)
provide('tt-config', props)

function camelToKebab(str: string): string {
  return str.replace(/([A-Z])/g, '-$1').toLowerCase()
}
</script>

<style lang="scss">
.tt-config-provider {
  // 确保 CSS Variables 向下级联
  // 小程序端需要此设置让变量穿透到子组件
}
</style>
```

## 4. 关键实现细节

### 4.1 小程序样式隔离问题

小程序自定义组件默认 `styleIsolation: 'apply-shared'`，外部定义的 CSS Variables 无法穿透到子组件。

**解决方案**：ConfigProvider 通过 `:style` 内联方式注入 CSS Variables，而非 `<style>` 块定义。内联 style 会作为 DOM 属性向下继承，不受样式隔离限制。

```
ConfigProvider (style="--tt-primary: #171717; ...")
  └── TtButton (使用 var(--tt-primary) ✅ 可以读取)
       └── TtIcon (使用 var(--tt-primary) ✅ 可以读取)
```

### 4.2 CSS Variables 在各平台的继承行为

| 平台 | 通过 :style 注入 | 通过 class 注入 | page 选择器 |
|------|-----------------|----------------|------------|
| H5 | ✅ 向下继承 | ✅ 向下继承 | ✅ 等同 :root |
| 微信小程序 | ✅ 向下继承 | ⚠️ 受隔离限制 | ✅ 页面级有效 |
| 支付宝小程序 | ✅ 向下继承 | ⚠️ 受隔离限制 | ✅ 页面级有效 |
| App (WebView) | ✅ 向下继承 | ✅ 向下继承 | ✅ 页面级有效 |

**结论**：`:style` 内联注入是唯一在所有平台都可靠继承的方式。

### 4.3 页面间主题同步

由于 UniApp 中 provide/inject 不跨页面，需要额外的同步机制：

```
方案：模块级单例 + Storage 持久化

use-theme.ts 中的 themeMode 和 systemTheme 是模块级 ref，
所有 import 该模块的页面共享同一个响应式引用。

页面 A: import { useTheme } → 获得 resolvedTheme (ref)
页面 B: import { useTheme } → 获得同一个 resolvedTheme (ref)

当 A 页面调用 setTheme('dark')：
  → themeMode.value = 'dark'          (模块级 ref 变更)
  → uni.setStorageSync(...)           (持久化)
  → B 页面的 resolvedTheme 自动更新   (同一 ref)
  → ConfigProvider 的 :style 重新计算  (computed 触发)
  → 所有组件 CSS Variables 更新        (浏览器/运行时渲染)
```

**注意**：小程序中页面切换时旧页面可能被销毁重建。`onShow` 时主题状态通过模块级 ref 自动获取最新值，无需额外处理。但首次加载需调用 `initTheme()` 从 Storage 恢复。

### 4.4 rpx 在 CSS Variables 中的限制

**重要**：CSS Variables 的值在小程序中**不支持 rpx 单位的自动转换**。

```css
/* ❌ 不工作 — rpx 不会被转换 */
:root { --tt-radius: 12rpx; }
.btn { border-radius: var(--tt-radius); }

/* ✅ 方案 1：使用 px 单位 */
:root { --tt-radius: 6px; }

/* ✅ 方案 2：JS 中转换后注入 */
const radius = uni.upx2px(12) + 'px'
```

TT-UI 推荐方案：**Token 中使用 px 值**，通过工具函数提供 rpx → px 转换：

```typescript
// styles/tokens.ts
function rpx(value: number): string {
  // #ifdef H5
  return `${value}rpx`
  // #endif
  // #ifndef H5
  return `${uni.upx2px(value)}px`
  // #endif
}

export const lightTokens = {
  radius: rpx(12),
  // ...
}
```

## 5. 用户使用方式

### 5.1 基础用法（推荐）

```vue
<!-- 每个页面的根组件 -->
<template>
  <tt-config-provider>
    <view class="page-content">
      <tt-button>按钮</tt-button>
      <tt-switch v-model="isDark" @change="onThemeChange" />
    </view>
  </tt-config-provider>
</template>

<script setup>
import { useTheme } from 'tt-ui'

const { isDark, toggleTheme } = useTheme()

function onThemeChange() {
  toggleTheme()
}
</script>
```

### 5.2 配合 @uni-ku/root（全局方案，可选）

安装 `@uni-ku/root` 后，只需在 `App.ku.vue` 中包裹一次：

```vue
<!-- App.ku.vue -->
<template>
  <tt-config-provider>
    <KuRootView />
  </tt-config-provider>
</template>
```

此后所有页面自动继承主题配置，无需每页包裹。

### 5.3 局部主题覆盖

```vue
<template>
  <!-- 全局 light 主题 -->
  <tt-config-provider>
    <tt-card>Light Card</tt-card>

    <!-- 局部 dark 主题 -->
    <tt-config-provider theme="dark">
      <tt-card>Dark Card</tt-card>
    </tt-config-provider>
  </tt-config-provider>
</template>
```

### 5.4 自定义品牌色

```vue
<template>
  <tt-config-provider :theme-vars="brandTheme">
    <tt-button>品牌按钮</tt-button>
  </tt-config-provider>
</template>

<script setup>
const brandTheme = {
  primary: '#6366f1',
  primaryForeground: '#ffffff',
  radius: '16px'
}
</script>
```

## 6. 各方案最终选型对比

| 维度 | 方案 A 手动绑定 | 方案 B ConfigProvider | 方案 C @uni-ku/root |
|------|---------------|---------------------|-------------------|
| 侵入性 | 高（每页手动） | 中（每页包裹） | 低（一处配置） |
| 多端兼容 | ✅ | ✅ | ⚠️ 依赖插件兼容性 |
| 局部主题 | ❌ | ✅ | ✅ |
| 第三方依赖 | 无 | 无 | @uni-ku/root |
| 调试便利性 | ✅ | ✅ | ⚠️ 编译注入 |

**TT-UI 推荐组合**：

- **默认**：ConfigProvider（方案 B）— 无额外依赖，每个页面包裹一次
- **进阶**：ConfigProvider + @uni-ku/root（方案 C）— 文档中提供可选方案
- **系统 UI**：微信 DarkMode 配置 — 处理导航栏、TabBar 等系统组件

## 7. 实现优先级

| 阶段 | 内容 | 说明 |
|------|------|------|
| Phase 1 | `use-theme` composable + `lightTokens/darkTokens` | 主题状态管理核心 |
| Phase 1 | `ConfigProvider` 组件 | 主题注入的载体 |
| Phase 1 | Light / Dark 双主题 Token | shadcn 风格色值 |
| Phase 2 | 微信 DarkMode 系统配置 | 导航栏等系统 UI 适配 |
| Phase 2 | 品牌色自定义 API | `themeVars` 支持 |
| Phase 3 | @uni-ku/root 集成文档 | 可选的全局方案 |
| Phase 3 | 主题编辑器工具 | 可视化生成 Token |

## 8. 对原设计文档的修订建议

| # | 原设计 | 修订为 |
|---|--------|--------|
| 1 | 在 `:root` 定义 CSS Variables | 在 ConfigProvider 组件的 `:style` 中注入 |
| 2 | 通过 `.dark` class 切换主题 | 通过 ConfigProvider 的 `theme` prop 切换 |
| 3 | Token 使用 rpx 单位 | Token 使用 px 值（rpx 在 CSS Variables 中不转换） |
| 4 | App.vue provide/inject | 模块级单例 ref + Storage 持久化 |
| 5 | 缺少系统暗黑模式适配 | 补充 manifest.json + theme.json 配置 |
| 6 | Token 定义在 .scss 文件 | Token 定义在 .ts 文件，运行时通过 JS 注入 |
