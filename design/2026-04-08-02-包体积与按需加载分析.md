# TT-UI 包体积与按需加载分析

> 针对当前设计目录结构的包大小风险分析，及开源组件库按需加载方案对比

---

## 1. 问题背景

微信小程序硬性约束：

| 限制 | 大小 |
|------|------|
| 主包 / 单个分包 | ≤ 2MB |
| 所有分包合计 | ≤ 30MB |

一个中型小程序的主包通常已占用 800KB–1.2MB（业务代码 + 框架运行时）。组件库如果不做按需加载，可能直接撑爆主包。

---

## 2. 当前设计的包大小风险

### 2.1 问题一：总入口 index.ts 会导致全量引入

当前设计（`packages/index.ts`）：

```typescript
// packages/index.ts — 总入口
export { TtButton } from './components/tt-button'
export { TtInput } from './components/tt-input'
export { TtDialog } from './components/tt-dialog'
// ... 56 个组件全部导出
```

**风险**：如果用户通过 `import { TtButton } from 'tt-ui'` 引入，取决于构建工具的 tree-shaking 能力。UniApp 的编译器基于 Vite，理论上支持 ES Module tree-shaking，但需要满足以下前提：

1. `package.json` 必须声明 `"sideEffects": false`（当前设计中缺失）
2. 组件的样式不能通过全局 `@import` 引入（否则会被视为 side effect）
3. 各组件之间不能有隐式的交叉依赖

**当前设计缺失了以上全部 3 个前提条件。**

### 2.2 问题二：easycom 不等于 Tree Shaking

设计文档第 6.4 节声称"easycom 本身就是按需加载"。这需要精确区分：

| 机制 | easycom | ES Module Tree Shaking |
|------|---------|----------------------|
| 工作层级 | 编译时模板分析 | 构建时模块分析 |
| 判断依据 | 模板中是否出现 `<tt-xxx>` 标签 | JS 代码中是否 import |
| 未使用组件 | ✅ 不会被编译 | ✅ 不会被打包 |
| JS 依赖树 | ❌ 不分析 | ✅ 深度分析 |
| 样式处理 | ❌ 不处理 | ❌ 需要配合 sideEffects |
| 分包场景 | ⚠️ npm 组件默认进主包 | ⚠️ 同样问题 |

**结论**：easycom 在组件维度确实是按需的（未使用的组件文件不参与编译），但它不处理组件内部的 JS 依赖树和样式体积问题。对于组件库体积优化，两者需要配合使用。

### 2.3 问题三：SCSS 注入导致样式膨胀

UniApp 的 `uni.scss` 有特殊行为——编译器会在**每个使用 `lang="scss"` 的组件**开头自动注入 `uni.scss` 的全部内容。

如果在 `uni.scss` 中引入组件库样式：

```scss
// ❌ 错误做法 — 会导致样式在每个组件中重复
@import 'tt-ui/packages/styles/tokens.scss';
@import 'tt-ui/packages/styles/base.scss';
```

假设 `tokens.scss` + `base.scss` = 5KB，项目有 40 个使用 scss 的页面/组件：
**膨胀结果 = 5KB × 40 = 200KB 纯冗余样式**

### 2.4 问题四：npm 包 main 指向源码

当前 `package.json` 设计：

```json
{
  "main": "packages/index.ts",
  "module": "packages/index.ts"
}
```

**问题**：
- 消费者必须有 TS 编译能力（增加构建时间）
- 源码中的 SCSS 需要消费者安装 sass 依赖
- 无法利用预编译优化（如压缩、dead code elimination）
- 小程序端不支持直接运行 TS

### 2.5 问题五：分包场景组件被打入主包

当使用 easycom + npm 安装时，`node_modules` 中的组件代码**默认被编译到主包的 vendor.js 中**，即使该组件只在某个分包页面使用。

这意味着 56 个组件如果有任何一个被主包页面引用，整个组件库的公共依赖（utils、composables）都会进入主包。

## 3. 开源组件库的按需加载方案对比

### 3.1 wot-design-uni

| 维度 | 方案 |
|------|------|
| 安装方式 | uni_modules（推荐）/ npm |
| 按需引入 | `@uni-helper/vite-plugin-uni-components` + `WotResolver` |
| 组件体积 | 整包 ~1.25MB，components 目录 ~30.8KB（gzip） |
| 样式方案 | CSS Variables 定义在组件内部 `<style>` 中，不依赖全局注入 |
| 特点 | 每个组件完全独立，无交叉 import |

**关键设计**：每个组件的样式写在 `.vue` 文件的 `<style>` 块中（非外部 SCSS 文件），避免了样式抽取和重复注入问题。

### 3.2 Vant Weapp

| 维度 | 方案 |
|------|------|
| 安装方式 | npm |
| 按需引入 | `vant-tree-shaking` 工具（打包后删除未用组件目录） |
| 问题 | 微信小程序构建 npm 时会拷贝整个库到 miniprogram_npm |
| 特点 | 后处理方式，不够优雅但有效 |

### 3.3 uni-ui（官方）

| 维度 | 方案 |
|------|------|
| 安装方式 | uni_modules（推荐） |
| 按需引入 | easycom 默认规范，自动按需 |
| 特点 | 每个组件是独立的 uni_module，可单独安装/卸载 |

### 3.4 方案对比总结

| 方案 | 按需粒度 | 样式处理 | 分包友好 | 复杂度 |
|------|---------|---------|---------|--------|
| easycom (uni_modules) | 组件级 ✅ | 组件内置 ✅ | ✅ | 低 |
| easycom (npm) | 组件级 ✅ | 需额外处理 ⚠️ | ❌ 默认进主包 | 中 |
| vite-plugin-uni-components | 组件级 ✅ | 需 sideEffects ⚠️ | ❌ 默认进主包 | 中 |
| 手动 import | 组件级 ✅ | 手动管理 | ❌ 默认进主包 | 高 |
| vant-tree-shaking（后处理） | 组件级 ✅ | 自动 ✅ | ❌ | 低 |

## 4. TT-UI 包体积优化方案

### 4.1 推荐架构：双轨分发（npm + uni_modules）

```
tt-ui/
├── packages/                # 源码（开发用）
│   ├── components/
│   │   ├── tt-button/
│   │   │   ├── tt-button.vue
│   │   │   ├── props.ts
│   │   │   └── index.ts
│   │   └── ...
│   └── ...
│
├── dist/                    # npm 构建产物
│   ├── es/                  # ESM 格式（Tree Shaking 友好）
│   │   ├── tt-button/
│   │   │   ├── index.mjs
│   │   │   └── style.css
│   │   ├── tt-input/
│   │   └── ...
│   ├── lib/                 # CJS 格式
│   └── resolver.mjs         # 自定义 Resolver
│
└── uni_modules/             # uni_modules 格式产物
    └── tt-ui/
        ├── components/
        │   ├── tt-button/
        │   │   └── tt-button.vue
        │   └── ...
        ├── package.json
        └── readme.md
```

### 4.2 组件独立性原则（最关键）

**每个组件必须满足以下条件：**

```
tt-button/
├── tt-button.vue       # 模板 + 脚本 + 样式（全部自包含）
├── props.ts            # Props 定义
└── index.ts            # 导出入口
```

1. **样式内聚**：所有样式写在 `.vue` 的 `<style>` 块中，不 `@import` 外部 SCSS 文件
2. **CSS Variables 引用方式**：组件通过 `var(--tt-primary)` 引用变量，变量定义在 `App.vue` 或 `ConfigProvider` 中
3. **零交叉依赖**：组件 A 不直接 import 组件 B 的内部文件。如需复用，通过 `composables/` 公共函数
4. **工具函数按文件 import**：`import { isDef } from '../utils/shared'` 而非 `import { isDef } from '../utils'`

### 4.3 CSS Variables 注入策略

```
❌ 错误：在 uni.scss 中引入 tokens
   → 每个 .vue 组件都会重复注入

✅ 正确方案 A：在 App.vue 中引入（推荐）

<style>
@import 'tt-ui/styles/tokens.css';  /* 只引入一次 */
</style>

✅ 正确方案 B：ConfigProvider 组件注入

<tt-config-provider :theme="themeTokens">
  <!-- 页面内容 -->
</tt-config-provider>
```

**体积对比**：

| 方案 | tokens.css 体积 | 40 个页面总注入 |
|------|----------------|---------------|
| uni.scss 注入 | 3KB × 40 = **120KB** | 120KB 冗余 |
| App.vue 引入 | 3KB × 1 = **3KB** | 0 冗余 |
| ConfigProvider | ~0（JS 动态设置） | 0 冗余 |

### 4.4 package.json 正确配置

```json
{
  "name": "tt-ui",
  "version": "0.1.0",
  "main": "dist/lib/index.js",
  "module": "dist/es/index.mjs",
  "types": "dist/es/index.d.ts",
  "sideEffects": [
    "dist/es/*/style.css"
  ],
  "exports": {
    ".": {
      "import": "./dist/es/index.mjs",
      "require": "./dist/lib/index.js",
      "types": "./dist/es/index.d.ts"
    },
    "./es/*": "./dist/es/*",
    "./lib/*": "./dist/lib/*",
    "./resolver": "./dist/resolver.mjs",
    "./styles/*": "./dist/styles/*"
  },
  "peerDependencies": {
    "vue": "^3.2.0"
  }
}
```

**关键字段说明**：

- `"sideEffects": ["dist/es/*/style.css"]` — 告诉打包工具只有 CSS 文件有副作用，其他模块可安全 tree-shake
- `"module"` 指向 ESM 产物 — Vite 优先使用此入口，天然支持 tree-shaking
- 不再指向 `.ts` 源码 — 消费者无需 TS 编译

### 4.5 Resolver 实现

为 `@uni-helper/vite-plugin-uni-components` 提供自定义 Resolver：

```typescript
// packages/resolver.ts
export interface TtUIResolverOptions {
  importStyle?: boolean | 'css' | 'scss'
}

export function TtUIResolver(options: TtUIResolverOptions = {}) {
  const { importStyle = true } = options

  return {
    type: 'component' as const,
    resolve: (name: string) => {
      if (name.startsWith('Tt')) {
        const partialName = name.slice(2)
        const kebabName = partialName
          .replace(/([A-Z])/g, '-$1')
          .toLowerCase()
          .slice(1)

        return {
          name: 'default',
          from: `tt-ui/es/tt-${kebabName}/index.mjs`,
          sideEffects: importStyle
            ? [`tt-ui/es/tt-${kebabName}/style.css`]
            : []
        }
      }
    }
  }
}
```

### 4.6 分包优化策略

解决 npm 组件默认进主包的问题：

**方案 A：manifest.json 配置分包优化（推荐）**

```json
{
  "mp-weixin": {
    "optimization": {
      "subpackages": true
    }
  }
}
```

此配置让 UniApp 编译器将仅在分包中使用的组件代码，打包到对应分包而非主包。

**方案 B：分包 uni_modules**

在分包根目录放置独立的 `uni_modules`，仅包含该分包需要的组件：

```
src/
├── pages/              # 主包
├── pagesA/             # 分包A
│   ├── uni_modules/    # 分包A专用组件
│   │   └── tt-ui/
│   │       └── components/
│   │           └── tt-chart/   # 仅分包A用的大型组件
│   └── pages/
└── pagesB/             # 分包B
```

**方案 C：lazyCodeLoading 配合**

在 `app.json` / `manifest.json` 中启用：

```json
{
  "mp-weixin": {
    "lazyCodeLoading": "requiredComponents"
  }
}
```

此配置让微信小程序运行时仅加载当前页面引用的组件代码，进一步减少内存占用。

## 5. 体积预估

基于 wot-design-uni 的实际数据推算：

| 指标 | wot-design-uni (80+ 组件) | TT-UI 预估 (56 组件) |
|------|--------------------------|---------------------|
| npm 包总大小 | ~1.25 MB | ~800 KB |
| components 目录 (gzip) | ~30.8 KB | ~20 KB |
| 单组件平均大小 (gzip) | ~0.4 KB | ~0.4 KB |
| 全量引入编译产物 | ~150-200 KB | ~100-140 KB |
| 10 组件按需引入 | ~15-25 KB | ~15-25 KB |

**结论**：只要正确实现按需引入，使用 10 个组件约占 15-25KB (gzip)，对 2MB 主包限制影响可控。

## 6. 构建流程设计

```
源码 (packages/)
  │
  ├── vite build --lib (ESM)
  │   → dist/es/
  │     ├── tt-button/
  │     │   ├── index.mjs      # 编译后的 JS
  │     │   ├── index.d.ts     # 类型声明
  │     │   └── style.css      # 提取的样式
  │     └── ...
  │
  ├── vite build --lib (CJS)
  │   → dist/lib/
  │
  ├── 脚本: 生成 uni_modules 格式
  │   → uni_modules/tt-ui/
  │     └── components/
  │         ├── tt-button/tt-button.vue  (单文件组件, 样式内嵌)
  │         └── ...
  │
  └── 脚本: 生成类型声明
      → dist/es/index.d.ts
      → dist/global.d.ts
```

### 6.1 Vite 库模式构建配置

```typescript
// vite.config.build.ts
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  build: {
    lib: {
      entry: 'packages/index.ts',
      formats: ['es']
    },
    rollupOptions: {
      external: ['vue', '@dcloudio/uni-app'],
      output: {
        preserveModules: true,         // 保持模块结构，tree-shaking 友好
        preserveModulesRoot: 'packages',
        entryFileNames: '[name].mjs',
        dir: 'dist/es'
      }
    },
    cssCodeSplit: true                  // 每个组件生成独立 CSS
  }
})
```

**关键配置**：`preserveModules: true` — 保持源码的模块结构，使打包工具能精确 tree-shake 到单个组件级别。

## 7. 对设计文档的修改建议

| # | 原设计 | 建议修改 |
|---|--------|---------|
| 1 | `"main": "packages/index.ts"` | 改为指向编译产物 `"main": "dist/lib/index.js"` |
| 2 | 无 sideEffects 配置 | 添加 `"sideEffects": ["dist/es/*/style.css"]` |
| 3 | 仅 npm 分发 | 增加 uni_modules 分发渠道 |
| 4 | easycom 作为按需引入方案 | easycom + vite-plugin-uni-components 双方案 |
| 5 | SCSS token 文件需全局引入 | CSS Variables 在 App.vue 一次性引入 |
| 6 | 缺少构建产物设计 | 补充 dist/ 目录结构和构建流程 |
| 7 | 样式可能依赖外部 SCSS | 组件样式完全自包含在 .vue 文件中 |

## 8. 总结

当前设计的核心风险不是"组件太多导致包大"，而是**缺少正确的按需加载基础设施**：

1. **没有 sideEffects 声明** → 打包工具无法安全 tree-shake
2. **package.json 入口指向源码** → 增加编译成本，且消费者需要额外配置
3. **样式注入方式未规划** → 可能导致 SCSS 膨胀
4. **仅 npm 分发** → 丢失 uni_modules 的天然按需优势
5. **缺少分包策略** → 所有组件默认进主包

只要修正以上 5 个问题，56 个组件的库在正确按需引入下，对用户项目的体积影响可以控制在 **15-25KB (gzip)** 级别，完全在可接受范围内。
