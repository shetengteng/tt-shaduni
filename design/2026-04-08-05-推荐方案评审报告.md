## Dimension B — 假设挑战

### B1. ConfigProvider `:style` 继承的前提条件

**原文声称**："`:style` 内联是唯一在所有平台都可靠继承的方式"

**挑战**：微信小程序自定义组件默认 `styleIsolation: 'apply-shared'`，在此模式下 CSS Variables 可以继承。但如果某个组件设置了 `styleIsolation: 'isolated'`，变量继承会断裂。

**建议**：在组件开发规范中增加硬性约束：

```
所有 TT-UI 组件不得设置 options.styleIsolation 为 'isolated'
```

### B2. 构建配置不完整

**原文**：第 3.3 节 Vite 构建配置仅包含 Rollup 选项

**挑战**：缺少 `@vitejs/plugin-vue` 插件。`.vue` SFC 文件无法被 Rollup 直接处理，需要 Vue 编译插件。当前配置直接运行会报错。

**建议**：补充完整的构建插件链：

```typescript
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
  build: { /* 原有配置 */ }
})
```

### B3. initTheme() 调用时机未明确

**原文**：`use-theme.ts` 使用模块级 `ref()` + `initTheme()` 函数

**挑战**：冷启动时需要从 Storage 恢复主题状态，但文档没有说明 `initTheme()` 何时调用。如果遗漏，首次进入页面会闪白（light 默认）再切换到用户保存的主题。

**建议**：在 ConfigProvider 的 `setup()` 中自动调用 `initTheme()`，确保首次渲染前主题已就绪。

### B4. SCSS 是否必要

**原文**：组件使用 `<style lang="scss">`

**挑战**：如果组件内部只使用 `var(--tt-xxx)` 引用 CSS Variables，并不需要 SCSS 的 mixin/variables/nesting 等特性。使用 `lang="scss"` 会带来额外的编译依赖和 `uni.scss` 注入风险。

**建议**：评估去掉 SCSS、改用纯 CSS 或 PostCSS nesting 的可行性。如果决定保留 SCSS，必须严格确保不在 `uni.scss` 中引入任何 TT-UI 样式。

## Dimension C — 视角盲点

### C1. 组件间依赖链未处理

**场景**：Button 内部使用 `<tt-loading>` 组件（loading 状态时显示加载动画）

**问题**：Resolver 将 `TtButton` 映射到 `tt-ui/es/tt-button/index.mjs`，但不会自动引入 `tt-loading` 的 JS 和 CSS。两种解决路径：

| 方案 | 说明 | 风险 |
|------|------|------|
| 构建时打入依赖 | Button bundle 中包含 Loading | 如果 Dialog 也用 Loading，Loading 被重复打包 |
| Resolver 维护依赖图 | 自动补充 `sideEffects` | 需要额外维护，但体积最优 |

**建议**：参考 wot-design-uni 的做法，在 Resolver 中维护依赖映射表。

### C2. 无障碍（a11y）完全缺失

shadcn/ui 的核心卖点之一是基于 Radix Primitives 的无障碍支持。TT-UI 参考了视觉风格，但未提及 ARIA 属性：

- Button: `role="button"`, `aria-disabled`
- Dialog: `role="dialog"`, `aria-modal`, 焦点陷阱
- Input: `aria-label`, `aria-describedby`
- Tabs: `role="tablist"`, `role="tab"`, `aria-selected`

**建议**：至少在 MVP 10 个组件中加入基础 ARIA 属性。H5 端直接生效，小程序端部分支持。

### C3. Token 命名转换规则未显式化

TS Token 使用 camelCase（`primaryForeground`），CSS Variable 需要 kebab-case（`--tt-primary-foreground`）。转换函数在主题切换设计文档中存在，但推荐方案汇总中未体现。

**建议**：在第 4 节补充转换规则说明或引用。

### C4. 缺少错误边界 / 降级策略

如果 ConfigProvider 未被包裹，组件内的 `var(--tt-primary)` 读不到值会怎样？

- H5: 元素使用 CSS 的 initial value（通常透明/黑色）
- 小程序: 同上

**建议**：为核心 Token 提供 CSS fallback：

```css
background: var(--tt-primary, #171717);
```

## Dimension D — 风险压测

### D1. ConfigProvider 嵌套时的 Token 合并策略

**场景**：

```vue
<tt-config-provider>                    <!-- 注入 22 个 light token -->
  <tt-config-provider theme="dark">    <!-- 注入 22 个 dark token -->
    <tt-config-provider :theme-vars="{ primary: '#6366f1' }">  <!-- ? -->
    </tt-config-provider>
  </tt-config-provider>
</tt-config-provider>
```

最内层只想覆盖 `primary`，但当前设计会注入什么？如果是全量 lightTokens + `{ primary: '#6366f1' }`，会覆盖掉中间层的 dark 模式变量。

**建议**：嵌套 ConfigProvider 应实现 diff merge — 仅注入与父级不同的变量，而非全量覆盖。

### D2. Token 命名是公开 API

如果 `v0.2` 将 `--tt-primary` 改名为 `--tt-brand`，所有用户的自定义主题（通过 `theme-vars` 或 CSS 覆盖）都会失效。

**建议**：声明 Token 稳定性约定：

- `0.x` 阶段：Token 名称可能变更，发布时在 CHANGELOG 中标注
- `1.0` 起：Token 名称变更视为 breaking change

### D3. 首次实施失败的回退方案

如果 MVP 验证发现 CSS Variables 在某个目标平台（如支付宝小程序）的继承行为不符合预期，回退成本是什么？

- **低成本回退**：仅移除 ConfigProvider 的 `:style` 注入，改为每个组件在 `setup()` 中自行读取主题并通过 `:style` 设置
- **这意味着**：放弃优雅的级联继承，但主题系统仍然可用

**建议**：在 MVP 中同时测试微信 + 支付宝 + H5 三个平台，尽早发现兼容性问题。
